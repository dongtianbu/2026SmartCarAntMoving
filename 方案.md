# 技术方案

## 任务判断

- 当前赛题核心不是单纯识别，而是“识别 -> 接近 -> 接触 -> 推动 -> 协同 -> 分类送区”的整链路闭环。
- 当前视觉端已经实现纯色沙包检测并通过串口输出目标框。
- 当前主控侧已经实现根据视觉框中心做 PD 居中，但还没有进入推进与协同阶段。

## 分目标技术方案

### 推沙包

- 继续使用传统视觉纯色检测。
- 控制目标点从屏幕正中心改为下偏中心，让小车在横向对齐后主动前逼。
- 主控流程采用状态机：`SEARCH -> ALIGN_X -> APPROACH -> CONTACT -> PUSH_STRAIGHT -> FINISH/RECOVER`。
- 接触判定使用目标框底边、面积、宽高变化联合判断。
- 接触后从视觉居中切换为 `IMU` 航向保持 + 编码器定距直推。
- 双车协同时采用平行双推。

### 推网球

- 优先采用传统视觉：绿色阈值 + 圆形约束 + 宽高比约束。
- 难点主要在控制和机构，不在识别。
- 双车采用浅 `V` 形包夹，小速度闭环推进，避免球从中间滚走。
- 推球速度低于推沙包，角速度修正也要更柔和。

### 推小熊

- 小熊不建议继续走纯颜色方案，推荐单独使用轻量 AI 检测。
- 采用混合视觉路线：
  - 沙包：传统视觉
  - 网球：传统视觉
  - 小熊：AI 检测/分类
- 推小熊时应尽量作用于下半身，避免顶到上半身后姿态失稳。
- 推进速度要更低，接触后优先依赖 IMU 保向。

## 接下来的实现路线

1. 先打通单车沙包完整闭环。
   - 从现有 `PD 居中` 升级为 `SEARCH / ALIGN / APPROACH / CONTACT / PUSH / RECOVER` 状态机。

2. 接入姿态与里程闭环。
   - 使用现有 `IMUVertical.py` 和 `MotorEncoder.py`，在接触后执行航向锁定和定距推进。

3. 扩展视觉串口协议。
   - 复用 `idx` 作为类别号：
   - `0`: 红沙包
   - `1`: 蓝沙包
   - `2`: 网球
   - `3`: 小熊
   - `255`: 无目标

4. 实现双车协同。
   - 采用“一主一从”架构：
   - 主车负责视觉、目标选择、阶段管理。
   - 从车负责跟随主车阶段和相对位姿保持。

5. 最后实现决赛分类送区。
   - 重点补“按类别选择推送方向和目标区域”，而不是单纯增加识别种类。

## 最推荐方案

- 双车协同状态机 + 传统视觉为主 + 小熊单独 AI。

原因：

- 最大化复用当前已经跑通的沙包识别与 PD 居中成果。
- 研发风险集中在真正短板：推进状态机、接触判定、双车同步。
- 避免过早把全部物体识别都压到 AI，导致训练、部署、调参同时增加复杂度。
- 与当前代码现状最匹配：传统视觉已成型，AI 链路暂未真正投入使用。
