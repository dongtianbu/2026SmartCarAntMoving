# 当前代码控制逻辑说明

本文档解释当前目录下这套代码在“视觉识别色块并通过 PD 控制小车居中”场景下的完整控制逻辑，重点说明：

- 主程序如何组织控制流程
- `ColorTrace.py` 如何把色块位置转换为底盘运动指令
- `MotorControl.py` 如何把 `vx / vy / omega` 转换为三个电机占空比
- 为什么目前在“接近居中”时会出现轨迹不可预测

---

## 1. 整体控制链路

当前代码的主链路是：

1. 视觉板识别色块
2. 视觉板通过串口发送目标框
3. 下位机解析目标框，得到色块中心 `(cx, cy)`
4. 计算目标中心和屏幕中心之间的误差 `(err_x, err_y)`
5. 对 `err_x` 和 `err_y` 分别做 PD
6. 把 PD 输出当成底盘平移速度 `vx / vy`
7. 将 `vx / vy` 映射为三个轮子的 duty
8. 将 duty 输出给三轮全向底盘

这条链路对应的主要文件是：

- [main.py](/D:/2026_SmartCar/PersonalCode/main.py)
- [ColorTrace.py](/D:/2026_SmartCar/PersonalCode/ColorTrace.py)
- [MotorControl.py](/D:/2026_SmartCar/PersonalCode/MotorControl.py)
- [MCXVisionUsart.py](/D:/2026_SmartCar/PersonalCode/MCXVisionUsart.py)

---

## 2. `main.py` 的控制逻辑

`main.py` 是主控入口，职责是：

1. 初始化视觉串口 `MCXVisionUsart`
2. 初始化无线串口 `WirelessUsart`
3. 创建 `ColorTraceController`
4. 在主循环中不断调用 `color_trace.update(...)`
5. 把 update 的结果打印出来，便于调试

### 2.1 主循环在做什么

主循环每次执行时：

1. 调用 `color_trace.update(motor_1, motor_2, motor_3)`
2. 如果有返回值，就说明控制器完成了一轮“读视觉 -> 算误差 -> 算 PD -> 算 duty -> 输出电机”
3. 如果当前帧有目标，就把目标中心和框尺寸通过无线串口发出去
4. 定期打印调试信息：
   - `err_x / err_y`
   - `filtered_cx / filtered_cy`
   - `correction_x / correction_y`
   - `raw_duty`
   - `target_duty`
   - `duty`

所以 `main.py` 本身几乎不做控制计算，它只是：

- 负责参数配置
- 负责周期性调用控制器
- 负责调试输出

真正的控制逻辑主要在 `ColorTrace.py` 和 `MotorControl.py`

---

## 3. `ColorTrace.py` 的控制逻辑

`ColorTraceController` 的职责是：

1. 从视觉串口接收目标框
2. 计算目标位置误差
3. 对误差做滤波
4. 用两个 PD 控制器分别算出 X/Y 方向修正量
5. 把修正量变成底盘平移命令
6. 进一步变成三个电机的目标 duty

---

## 4. 视觉数据是怎么变成误差的

视觉板通过串口给下位机发送一个 11 字节的数据帧：

`[AA][idx][x1][y1][x2][y2][FF]`

下位机在 `MCXVisionUsart.parse_frame()` 中解析出：

- `x1, y1`
- `x2, y2`
- `center_x`
- `center_y`

控制器默认使用的逻辑屏幕坐标是：

- 宽 `160`
- 高 `120`

因此屏幕中心固定为：

- `CENTER_X = 80`
- `CENTER_Y = 60`

在 `ColorTraceController._frame_to_state()` 中，误差定义为：

```python
err_x = CENTER_X - cx
err_y = CENTER_Y - cy
```

含义是：

- 如果色块在屏幕右边，`cx > 80`，那么 `err_x < 0`
- 如果色块在屏幕左边，`cx < 80`，那么 `err_x > 0`
- 如果色块在屏幕下边，`cy > 60`，那么 `err_y < 0`
- 如果色块在屏幕上边，`cy < 60`，那么 `err_y > 0`

这一步把视觉问题转换成了“目标离中心还有多远”。

---

## 5. 为什么要先滤波

在 `_filter_center()` 中，代码会对 `cx, cy` 做一阶低通滤波：

```python
filtered += alpha * (current - filtered)
```

作用是：

- 抑制视觉框抖动
- 让车不要因为单帧噪声突然改变方向

缺点是：

- 滤波后有延迟
- 接近中心时，如果框本身在抖，滤波结果可能来回穿过中心

这会直接影响后面的 PD 输出方向。

---

## 6. PD 居中控制是怎么做的

当前不是一个二维统一控制器，而是两个独立的单轴 PD：

- `x_pid` 负责左右纠偏
- `y_pid` 负责前后纠偏

计算方式在 `PID.compute()`：

```python
error = target - current
output = kp * error + kd * (error - last_error)
```

而调用方式是：

```python
self.correction_x = self.x_pid.compute(0, err_x)
self.correction_y = self.y_pid.compute(0, err_y)
```

代入后得到：

```python
correction_x ≈ -(kp_x * err_x + kd_x * d_err_x)
correction_y ≈ -(kp_y * err_y + kd_y * d_err_y)
```

也就是说：

- `err_x` 为正，`correction_x` 倾向于负
- `err_x` 为负，`correction_x` 倾向于正
- `err_y` 为正，`correction_y` 倾向于负
- `err_y` 为负，`correction_y` 倾向于正

这本质上是“把目标拉回中心”的负反馈。

### 6.1 死区

只有当下面条件成立时，系统才认为已经居中：

```python
abs(err_x) <= dead_zone and abs(err_y) <= dead_zone
```

也就是一个矩形死区，不是圆形死区。

进入死区后：

- `correction_x = 0`
- `correction_y = 0`
- `moving = False`
- 重置 `x_pid` 和 `y_pid`

这意味着当前控制是“有误差就动，进死区就立即停”。

### 6.2 最小修正速度

为了防止“PD 输出太小，车根本起不来”，代码又加入了：

```python
_enforce_min_correction()
```

逻辑是：

- 只要目标已经不在死区内
- 如果 `correction_x / correction_y` 的量级太小
- 就把整个修正向量放大到 `min_correction_speed`

这一步解决了“不动”的问题，但它也引入了一个副作用：

- 一旦刚离开死区，修正量不是连续增加，而是直接跳到一个最小值

这会让中心附近的运动呈现出明显的不连续性。

## 7. 从 PD 输出到底盘平移命令

当控制器认为当前需要运动时，会调用：

```python
MotorControl.vector_to_duty(
    self.correction_x,
    -self.correction_y,
    max_duty=active_max_duty,
    min_duty_start=0,
)
```

这里有两个关键点：

### 7.1 `correction_x` 直接当作 `vx`

表示底盘在左右方向上的目标速度。

### 7.2 `correction_y` 被取反

也就是：

```python
vy = -correction_y
```

原因是图像坐标和底盘坐标方向不一致，需要在这里做一次坐标变换。

所以当前控制本质上是：

- 视觉误差 `err_x / err_y`
- 经过 PD 得到 `correction_x / correction_y`
- 转换为底盘平移速度 `vx / vy`

注意：这里没有对 `omega` 做控制，也就是说当前系统没有显式控制车体航向。

---

## 8. `MotorControl.py` 的电机控制逻辑

`MotorControl.py` 的职责是把底盘速度转换成三个电机的 duty。

### 8.1 三轮全向底盘混控

函数：

```python
chassis_vector_to_wheel_speed(vx, vy, omega=0)
```

使用矩阵 `WHEEL_MIXING`：

```python
(
    (-0.5, -cos(pi/6)),
    ( 1.0,  0.0),
    (-0.5,  cos(pi/6)),
)
```

它的含义是：

- 底盘的平移速度 `vx / vy`
- 经过几何混控
- 被分解成三个轮子的转速需求 `v1 / v2 / v3`

这一步决定了：

- 同样的视觉误差方向
- 会如何映射成三个轮子的正反转和幅值

### 8.2 速度到 duty 的映射

函数：

```python
V1V2V3DSpeedToDuty(v1, v2, v3, max_duty, min_duty_start)
```

逻辑如下：

1. 先把三轮速度限制在 `MAX_SPEED`
2. 再线性映射成 duty
3. 如果非零 duty 太小，小于 `min_duty_start`
4. 就整体放大，确保轮子能起转
5. 如果放大后超过 `max_duty`
6. 再整体缩回去

这意味着电机控制不是严格线性的，而是带有“起步门槛补偿”的分段非线性映射。

### 8.3 当前控制器里的二次非线性

当前在 `ColorTrace.py` 中，虽然调用 `vector_to_duty()` 时传的是：

```python
min_duty_start=0
```

但后面又经过了一层 `_bias_duty()`：

```python
self.target_duty = tuple(
    self._bias_duty(value, active_min_duty, active_max_duty)
    for value in raw_duty
)
```

也就是说：

1. 先算出一个原始轮速对应的 `raw_duty`
2. 只要某个轮子是非零
3. 就再把它偏置到一个最小有效区间

所以当前的轮子输出不是简单的“误差越小，duty 越小”，而是：

- 误差一旦非零
- 某些轮子就可能被顶到一个最低有效 duty

这也是中心附近轨迹不平滑的重要原因。

## 9. 为什么会出现“居中时轨迹不可预测”

这个问题不是单点原因，而是多种机制叠加。

### 9.1 矩形死区导致边界切换突兀

当前死区判定是：

```python
abs(err_x) <= dead_zone and abs(err_y) <= dead_zone
```

这相当于一个矩形框。

问题在于：

- 一旦坐标刚好在死区边缘来回跳
- 控制器就会在“完全停止”和“开始修正”之间切换

结果是：

- 有时停
- 有时又突然动一下

轨迹就会显得不连续。

### 9.2 视觉抖动引起修正方向翻转

接近中心时，色块中心点可能在：

- `(79, 60)`
- `(81, 60)`
- `(80, 59)`
- `(80, 61)`

之间来回跳。

这样会导致：

- `err_x` 正负来回变
- `err_y` 正负来回变

而 PD 输出方向也跟着翻转。

对于三轮全向底盘来说，方向稍微一变，三个电机的组合就完全不同，所以轨迹看起来会很随机。

### 9.3 最小修正速度放大了中心附近的小误差

当前加入了 `min_correction_speed`。

它的好处是：

- 小误差时也能动起来

它的坏处是：

- 明明只是刚离开死区一点点
- 系统却强制给出一个最小修正向量

所以中心附近的微小误差，会被放大成一次真实可见的底盘运动。

### 9.4 `_bias_duty()` 让轮子输出不再连续

只要某个轮子被判定为非零，它就会被拉到最小有效 duty 区间。

这会导致：

- 很小的目标速度
- 也可能触发明显的轮子动作

因此系统在中心附近不是真正“轻轻挪一点”，而更像“判断需要动后立刻给一脚最低有效油门”。

### 9.5 没有航向控制

当前控制只管 `vx / vy`，没有控制 `omega`。

所以一旦出现下面任意一种情况：

- 三个电机输出不完全一致
- 轮胎摩擦不一致
- 地面阻力不一致
- 车体机械装配存在偏差

车就会在平移的同时带一点旋转。

而视觉控制本身又没有显式抑制旋转，这种旋转会继续影响后续视觉误差，最终让轨迹看起来更不可预测。

### 9.6 中心附近是多重非线性叠加区

接近中心时，同时存在：

- 视觉滤波
- 死区切换
- PD 输出翻转
- 最小修正速度
- 最小有效 duty 偏置
- 三轮混控
- 无航向闭环

因此中心附近不是线性可预测区，而是一个很强的非线性区。

这就是为什么“远处追目标还比较正常，越接近中心越容易表现怪异”。

---

## 10. 当前控制逻辑的本质特点

可以把这套控制理解为：

- 上层是二维视觉误差闭环
- 中层是两个独立单轴 PD
- 下层是三轮全向底盘混控
- 电机侧带有最小起步补偿和占空比偏置

它的优点是：

- 结构简单
- 上手快
- 远距离追踪有效

它的缺点是：

- 中心附近不够连续
- 缺少姿态控制
- 非线性较多
- 轨迹稳定性依赖视觉抖动和机械一致性

---

## 11. 现阶段对“轨迹不可预测”问题的直接判断

如果只基于当前代码逻辑，不考虑硬件故障，那么小车居中时轨迹不可预测，最可能来自下面几项叠加：

1. 色块中心在死区边缘抖动
2. `min_correction_speed` 让微小误差被放大
3. `_bias_duty()` 让轮子一旦动作就不是很小的动作
4. 三轮底盘在没有 `omega` 闭环时会带出旋转分量
5. 左右/前后方向的 PD 是分离的，不是统一路径规划

换句话说，当前问题不是“PD 完全错了”，而是：

- 这套控制在大误差阶段能工作
- 但在接近中心时，控制连续性不够好

---

## 12. 一句话总结

当前代码的控制逻辑可以概括为：

**视觉框中心误差 -> 双轴 PD -> 底盘平移速度 -> 三轮混控 -> 占空比输出**

而“居中时轨迹不可预测”的根本原因是：

**中心附近同时存在死区切换、最小修正速度、最小 duty 偏置、视觉抖动和无航向控制，导致系统在小误差区呈现明显非线性。**
